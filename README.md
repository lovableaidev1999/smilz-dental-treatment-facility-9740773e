# Supabase Setup — IMPORTANT, READ BEFORE TOUCHING DB CODE

This project has an **unusual dual backend configuration**. Read this fully before
making changes to anything Supabase / `.env` / database related.

---

## TL;DR

- The **real database** is an **external Supabase project**: `eukymrxxmvkchxfpjjuz`
  (URL: `https://eukymrxxmvkchxfpjjuz.supabase.co`)
- The app talks to it via **hardcoded** URL + anon key in
  `src/integrations/supabase/client.ts`.
- **Lovable Cloud is also enabled** on this project, which spawned a second,
  empty Supabase project (`wleqgpofmfefvvksxkea`). This one is **not used** at
  runtime, but it **owns the auto-generated `.env` file** and the Lovable
  agent's database tools.
- **Do not edit `.env`** — it is overwritten by Lovable Cloud on every build.
- **Do not edit `src/integrations/supabase/client.ts`** unless you are
  intentionally repointing the app to a different Supabase project.

---

## Why two Supabase projects?

Historically the site was built on the external Supabase
(`eukymrxxmvkchxfpjjuz`) — all live data (services, page_layouts, blog_posts,
media, auth users) lives there.

At some point Lovable Cloud was enabled on the project. Once enabled:

- Lovable Cloud **cannot be disabled** on an existing project.
- The "Connect Supabase" button in the Lovable UI becomes **greyed out**, so
  the external Supabase cannot be re-connected through the official integration.
- Lovable Cloud provisions its own Supabase project (`wleqgpofmfefvvksxkea`)
  and writes its credentials to `.env` automatically on every build.

To keep the site working against the real database, the Supabase client was
patched to **ignore `.env`** and use hardcoded credentials for the external
project.

---

## How the app actually connects

`src/integrations/supabase/client.ts`:

```ts
// Hardcoded — these win over anything in .env
const SUPABASE_URL = "https://eukymrxxmvkchxfpjjuz.supabase.co";
const SUPABASE_PUBLISHABLE_KEY = "<anon key for eukymrxxmvkchxfpjjuz>";

export const supabase = createClient<Database>(SUPABASE_URL, SUPABASE_PUBLISHABLE_KEY);
```

Everything in the app — `useQuery` hooks, `SmartPage`, admin editors, blog,
auth, media library — flows through this client and therefore hits the
**external Supabase**.

---

## What `.env` contains (and why you should ignore it)

`.env` is regenerated by Lovable Cloud on every build:

```
VITE_SUPABASE_URL=https://wleqgpofmfefvvksxkea.supabase.co        # Cloud, unused
VITE_SUPABASE_PUBLISHABLE_KEY=...                                  # Cloud, unused
VITE_SUPABASE_PROJECT_ID=wleqgpofmfefvvksxkea                      # Cloud, unused
```

These point at the **empty** Lovable Cloud Supabase project. The app does not
read them at runtime because `client.ts` overrides them.

**Do not:**
- Manually edit `.env` (it will be overwritten).
- Add Supabase code that reads `import.meta.env.VITE_SUPABASE_*` directly —
  always import the `supabase` client from `@/integrations/supabase/client`.

---

## What this means for the Lovable agent

The Lovable agent's built-in Supabase tools
(`supabase--read_query`, `supabase--migration`, `supabase--project_info`,
`supabase--cloud_status`, `supabase--configure_auth`, etc.) are **hard-wired
to the Lovable Cloud project** (`wleqgpofmfefvvksxkea`), not the external one.

That means the agent **cannot**:
- Run SQL queries against the real database from chat.
- Run migrations against the real database from chat.
- Inspect auth users, RLS policies, or storage buckets of the real database.

When changes need to be made to the real Supabase, they must be done by:
1. **Code that runs in the app** using the hardcoded `supabase` client
   (e.g. an admin page, a one-off script wired into a button), or
2. **Manually in the external Supabase SQL editor / dashboard**:
   https://supabase.com/dashboard/project/eukymrxxmvkchxfpjjuz

---

## What about `src/integrations/supabase/types.ts`?

That file is auto-generated by Lovable from the **Cloud** Supabase project's
schema, so it will appear empty / out-of-sync with the real database. The app
casts around this with `as any` in places where the real schema diverges.

If TypeScript types ever need to match the real DB precisely, generate them
manually from the external Supabase using its CLI:

```bash
supabase gen types typescript --project-id eukymrxxmvkchxfpjjuz > src/integrations/supabase/types.ts
```

(But be aware Lovable may overwrite it.)

---

## The clean long-term fix

The only way to make `.env`, the agent's tools, and the app all point at the
same (real) Supabase is to **migrate the project to a new Lovable project
where Lovable Cloud is left disabled**, then connect the external Supabase
through the standard "Connect Supabase" integration. Until that happens,
the hardcoded override in `client.ts` is what keeps the site alive.
